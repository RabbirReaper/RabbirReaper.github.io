專案架構概述

RabbirVocab 採用前後端分離架構，但在生產環境下會合併部署:

技術棧

  • 後端框架: Express 5.x
  • TypeScript: 類型安全
  • 資料庫: MongoDB + Mongoose
  • Session: express-session + connect-mongo
  • 安全: Helmet, CORS
  • 建置工具: TypeScript Compiler (tsc)

核心檔案詳解

server/app.ts - 應用程式核心

這是 Express 應用的核心配置檔案，負責所有 middleware 設定和路由配置。

1. 模組導入與環境設定

為什麼需要這些模組?

  • express: Web 框架核心
  • cors: 處理跨域請求 (開發時前端 5173 → 後端 3000)
  • helmet: 安全性 headers
  • morgan: HTTP 請求日誌
  • path + fileURLToPath: ES Module 環境下獲取檔案路徑

為什麼需要手動獲取 __dirname?

  • 因為使用 ES Module ("type": "module")
  • ES Module 中沒有內建的 __dirname
  • 需要透過 import.meta.url 手動計算

2. 安全性 Middleware

為什麼只在非生產環境使用 Helmet?

  • Helmet 的 CSP (Content Security Policy) 會阻止靜態資源載入
  • 生產環境需要提供前端 HTML/CSS/JS，CSP 會造成問題
  • 開發環境可以啟用來提前發現安全問題

最佳實踐: 生產環境應該配置適當的 CSP 規則，而非完全關閉

3. CORS 配置

為什麼需要 CORS?

  • 開發模式: 前端 (5173) 向後端 (3000) 發請求會被瀏覽器阻擋
  • credentials: true: 允許跨域請求攜帶 Session Cookie

為什麼生產環境也保留 CORS?

  • 即使同源，保留 CORS 不會造成問題
  • 未來可能有多個前端域名，保持靈活性

4. Body Parser

功能說明:

  • express.json(): 解析 JSON 格式的請求體
  • express.urlencoded(): 解析 URL-encoded 格式 (表單提交)
  • limit: '2mb': 限制請求體大小，防止 DoS 攻擊
  • extended: true: 支援解析複雜的巢狀物件

5. Logger (開發環境)

為什麼只在開發環境?

  • 開發時需要看到每個請求的日誌
  • 生產環境應該用專業的日誌系統 (如 Winston, Bunyan)
  • morgan('dev'): 彩色輸出，易於閱讀

6. Session Middleware

Session 的作用:

  • 用戶登入狀態管理
  • Session 資料存儲在 MongoDB (透過 connect-mongo)
  • Cookie 自動管理，支援 HttpOnly 和 SameSite

7. API 路由

路由設計原則:

  • /health: 健康檢查端點，用於監控和負載均衡器
  • /api/*: 所有 API 都加上 /api 前綴，與前端路由區分

8. 靜態檔案服務 (生產環境)

這是最關鍵的部分，決定了生產環境如何提供前端:

逐行解析:

  1. path.resolve(__dirname, '..')

    • __dirname 是 dist/server/
    • .. 往上一層到 dist/
    • staticPath = d:\...\RabbirVocab\dist

  2. app.use(express.static(staticPath))

    • 提供 dist/ 下的靜態檔案
    • 自動處理 /assets/*, /favicon.ico 等
    • 如果找到檔案就直接返回，找不到就繼續下一個 middleware

  3. SPA Fallback Middleware

為什麼需要 SPA Fallback?

Vue Router 使用 HTML5 History 模式:

  • 使用者訪問 /login → 應該返回 index.html
  • Vue Router 在客戶端解析 /login 並顯示對應元件
  • 如果沒有 fallback，伺服器會返回 404

為什麼要檢查 /api?

  • 避免 API 404 也返回 HTML
  • API 應該返回 JSON 錯誤訊息，不是 HTML
  • 透過 next() 讓請求繼續到 404 處理器

9. 錯誤處理

中間件順序很重要:

  1. 先處理 API 路由
  2. 生產環境: 提供靜態檔案 → SPA fallback
  3. 404 處理 (只針對 /api)
  4. 全域錯誤處理

server/server.ts - 入口點

功能說明:

  • dotenv.config(): 載入 .env 檔案的環境變數
  • startServer(): 啟動 Express 伺服器並連接資料庫

為什麼分離 app.ts 和 server.ts?

  • app.ts: 純粹的應用邏輯，可測試
  • server.ts: 入口點，負責啟動
  • 分離後可以在測試中 import { createApp } 而不啟動伺服器

package.json - 腳本配置

開發腳本

yarn dev

  • 啟動 Vite 開發伺服器 (前端)
  • 監聽 http://localhost:5173
  • 熱模組替換 (HMR)

yarn server:dev

  • 使用 tsx watch 直接執行 TypeScript
  • 檔案變更時自動重啟
  • 不需要手動編譯

yarn dev:all

  • 同時啟動前端和後端
  • 使用 concurrently 並行執行
  • 彩色輸出，易於區分

建置腳本

yarn server:build

  • 使用 TypeScript Compiler 編譯後端
  • 輸出到 dist/server/
  • 配置檔: tsconfig.server.json

yarn build:all

  • 先建置前端 (yarn build)
  • 再建置後端 (yarn server:build)
  • 前端輸出: dist/ (包含 index.html, assets/)
  • 後端輸出: dist/server/

生產腳本

yarn server:start

  • 執行已編譯的 JavaScript
  • 問題: 無法設定 NODE_ENV

yarn start:prod

  • 使用 cross-env 設定 NODE_ENV=production
  • 跨平台相容 (Windows/Linux/Mac)
  • 啟動生產模式伺服器
